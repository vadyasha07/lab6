| Параметр | Значення |
|----------|----------|

| **Відправка пошти** | Brevo API (REST) |

## Структура проєкту

```
lab6-server/
├── public/
│   ├── index.html      # Головна сторінка сайту
│   ├── style.css       # Стилі
│   └── script.js       # Frontend логіка (fetch)
├── server.js           # Головний файл сервера (Fastify)
├── .env                # Змінні середовища (конфігурація)
├── .gitignore          # Ігнорування файлів для Git
└── package.json        # Залежності проєкту
```

## Інструкція з запуску

### 1. Встановлення залежностей

```bash
cd lab6-server
npm install
```

### 2. Налаштування Brevo API

1. Зареєструйтесь на [brevo.com](https://www.brevo.com)
2. Перейдіть в **SMTP & API** → **API Keys**
3. Натисніть **Create a new API key**
4. Скопіюйте ключ
5. Вставте його у файл `.env` замість `your-brevo-api-key-here`

### 3. Налаштування email отримувача

У файлі `.env` замініть `your-email@example.com` на вашу реальну пошту.

### 4. Запуск сервера

```bash
node server.js
```

Сервер запуститься на `http://localhost:3000`

### 5. Тунелювання через Pinggy

1. Відкрийте **нове** вікно терміналу (не зупиняючи сервер)
2. Виконайте команду:

```bash
ssh -p 443 -R0:localhost:3000 a.pinggy.io
```

3. Pinggy згенерує публічний URL (наприклад, `https://xxxxx.a.pinggy.io`)
4. Надайте це посилання викладачу для перевірки

## Відповіді на контрольні питання

### 1. Що таке Node.js та чим його середовище відрізняється від виконання JavaScript у браузері?

**Node.js** — це кросплатформне середовище виконання JavaScript-коду, побудоване на рушії V8 від Google Chrome. Воно дозволяє виконувати JavaScript поза веббраузером.

**Відмінності від браузера:**
- У Node.js немає DOM, window, document — немає доступу до вебсторінок
- Є доступ до файлової системи, мережі, процесів ОС
- Використовується для серверної логіки, API, CLI-інструментів
- Має вбудовані модулі для роботи з HTTP, файлами, потоками тощо

### 2. Поясніть концепцію асинхронного та неблокуючого вводу/виводу (I/O) у Node.js.

**Асинхронний I/O** означає, що Node.js не чекає завершення операцій (читання файлу, запит до БД, мережевий запит). Замість очікування, він продовжує виконувати інший код, а результат обробляє через колбеки, проміси або async/await.

**Неблокуючий** — операції не "блокують" виконання програми. Наприклад, `fs.readFile()` читає файл у фоні, а програма продовжує працювати. Коли файл прочитано — викликається колбек або resolve промісу.

Це дозволяє обробляти тисячі з'єднань одночасно з мінімальними ресурсами.

### 3. Яка роль npm і для чого конкретно використовується файл package.json?

**npm (Node Package Manager)** — менеджер пакетів для Node.js. Дозволяє:
- Встановлювати сторонні бібліотеки (`npm install`)
- Керувати залежностями проєкту
- Запускати скрипти (`npm run`)

**package.json** — головний файл проєкту, який містить:
- Назву, версію, опис проєкту
- Список залежностей (`dependencies`, `devDependencies`)
- Скрипти для запуску (`scripts`)
- Точку входу (`main`)

### 4. Чому розробники зазвичай використовують веб-фреймворки замість створення серверів на "чистих" базових модулях Node.js?

Базовий модуль `http` вимагає ручного:
- Парсингу URL та тіла запиту
- Обробки MIME-типів
- Налаштування маршрутизації
- Обробки помилок

**Фреймворки** (Express, Fastify, Koa) надають:
- Зручний роутинг (`fastify.get('/path', handler)`)
- Вбудовану роздачу статики
- Автоматичний парсинг JSON
- Middleware систему
- Валідацію даних
- Готову обробку помилок

Це економить час і зменшує кількість помилок.

### 5. Що означає термін "маршрутизація" (роутинг) у контексті бекенд-розробки?

**Маршрутизація** — це механізм, який визначає, як сервер відповідає на HTTP-запити за певним шляхом (URL) та методом (GET, POST, PUT, DELETE тощо).

Приклад у Fastify:
```javascript
fastify.get('/', handler)        // GET /
fastify.post('/api/contact', handler)  // POST /api/contact
```

Кожен маршрут (route) прив'язує певний URL + HTTP-метод до функції-обробника.

### 6. Опишіть призначення API. Що таке API-ендпоінт (endpoint)?

**API (Application Programming Interface)** — інтерфейс програмування додатків, який дозволяє різним програмним системам спілкуватися між собою.

**API-ендпоінт** — це конкретний URL, на який клієнт може надіслати запит для отримання або відправки даних.

Приклад: `POST /api/contact` — ендпоінт, який приймає дані форми зворотного зв'язку.

### 7. Як веб-сервер роздає статичні файли і чому їх зазвичай розміщують в окремій папці?

**Статичні файли** — це файли, які сервер віддає без змін (HTML, CSS, JS, зображення).

Fastify використовує плагін `@fastify/static`, який:
- Мапить URL-шляхи до файлів у папці
- Автоматично визначає MIME-типи
- Кешує відповіді

**Окрема папка** (`public/`) — це:
- Чиста архітектура (відділення клієнтської та серверної частини)
- Безпека (сервер не віддає файли поза цією папкою)
- Зручність підтримки

### 8. У чому різниця між HTTP-методами GET та POST?

| GET | POST |
|-----|------|
| Отримує дані з сервера | Надсилає дані на сервер |
| Дані передаються в URL | Дані передаються в тілі запиту |
| Можна закладати (bookmark) | Не можна закладати |
| Кешується | Не кешується |
| Небезпечно для конфіденційних даних | Безпечніше для чутливих даних |

**Для форми** краще **POST**, тому що:
- Дані форми (ім'я, email, повідомлення) не мають бути в URL
- POST не кешується і не зберігається в історії браузера
- POST дозволяє надсилати більше даних

### 9. Чому необхідна валідація даних на стороні сервера, навіть якщо вона вже реалізована на фронтенді?

- **Фронтенд-валідацію можна обійти** — користувач може надіслати запит напряму через Postman/curl, минаючи браузер
- **Безпека** — шкідливі дані можуть пошкодити сервер або БД (SQL-ін'єкції, XSS)
- **Цілісність даних** — сервер має гарантувати, що дані відповідають очікуванням
- **Захист від ботів** — автоматизовані скрипти можуть надсилати невалідні дані

### 10. Що таке тунелювання під час локальної розробки і у яких випадках воно застосовується?

**Тунелювання** — створення захищеного "тунелю" від публічного URL до вашого локального сервера.

**Застосування:**
- Демонстрація проєкту викладачу/клієнту
- Тестування вебхуків (Stripe, Telegram Bot)
- Спільна робота над проєктом
- Тестування на мобільних пристроях

**Інструменти:** ngrok, Pinggy, localtunnel, Cloudflare Tunnel

### 11. Чому конфіденційні дані ніколи не можна "хардкодити" і як .env вирішує цю проблему?

**Хардкодинг** (вбудовування паролів у код) — це погана практика, тому що:
- Код потрапляє в Git — паролі стають доступні всім
- Важко змінювати конфігурацію для різних середовищ
- Ризик витоку при публікації на GitHub

**Файл `.env`** вирішує це:
- Зберігає секрети окремо від коду
- Додається в `.gitignore` — не потрапляє в репозиторій
- Легко змінювати для різних середовищ (dev, production)
- Читається через `process.env.VARIABLE_NAME`
